\examsection{一、选择题}{本大题共 12 小题，每小题 5 分，共 60 分。在每小题给出的四个选项中，只有一项是符合题目要求的。}

\qitem{1}{现代 NVIDIA GPU（如 H100）将大量计算单元组织为流多处理器（Streaming Multiprocessor, SM）。每个 SM 内含 CUDA Core（通用浮点/整数运算）、Tensor Core（矩阵乘累加 MMA）、Warp 调度器、Shared Memory（片上可编程缓存）和 Register File（线程私有寄存器）。线程以 Warp 为单位被调度（默认 32 线程/Warp），同一 Warp 内线程执行相同指令流（SIMT 模型）。关于上述结构，下列说法\textbf{正确}的是}{}
\fourchoices{每个 SM 仅包含一个 CUDA Core，所有线程在同一时刻只能串行执行一条指令}{每个 SM 拥有独立的 Warp 调度器，可将 32 个线程组成一个 Warp 进行 SIMT 调度；Warp 内线程若遇分支发散（Branch Divergence）会导致部分线程空闲}{SM 内的 Shared Memory 与 L2 Cache 在物理上是同一块存储，只是软件命名不同}{寄存器文件（Register File）位于 HBM 上，所有 SM 通过 PCIe 总线共享访问，延迟与 Global Memory 相同}

\qitem{2}{GPU 存储层级从快到慢依次为 Register $\to$ Shared Memory $\to$ L2 Cache $\to$ HBM（高带宽内存）。各级存储在容量、带宽和访问延迟上差异显著：Register 最快但每线程仅有数千字节；Shared Memory 为 SM 内共享、需程序员显式管理；L2 为芯片级统一缓存；HBM 容量最大（如 H100 为 80GB）但延迟最高。高效 Kernel 设计的核心目标之一，就是将数据尽可能长时间停留在高速存储层。关于带宽与延迟的排序，下列\textbf{正确}的是（从左到右：带宽递减 / 延迟递增）}{}
\fourchoices{Register $>$ Shared Memory $>$ L2 $>$ HBM}{HBM $>$ L2 $>$ Shared Memory $>$ Register}{Shared Memory $>$ Register $>$ HBM $>$ L2}{L2 $>$ HBM $>$ Register $>$ Shared Memory}

\qitem{3}{Roofline 模型（Williams et al., 2009）用算术强度 $I$（FLOP/Byte，即每从 HBM 搬运 1 字节数据能完成多少浮点运算）判断算子瓶颈。硬件 Ridge Point 定义为 $I^*=\dfrac{P_{\mathrm{peak}}}{B_w}$，其中 $P_{\mathrm{peak}}$ 为峰值算力、$B_w$ 为 HBM 峰值带宽。实际吞吐 $\approx\min(P_{\mathrm{peak}},\, I\cdot B_w)$：当 $I<I^*$ 时算力无法跑满，性能受带宽限制；当 $I>I^*$ 时带宽足够，性能受算力限制。大模型 Decode 阶段 GEMM 的 $M$ 维（批大小）很小，算术强度往往极低。当 $I < I^*$ 时，该算子处于}{}
\fourchoicestwo{Compute-bound（计算受限），应优先提升 Tensor Core 利用率}{Memory-bound（带宽受限），应优先减少 HBM 访问量或提高数据复用}{通信受限（Communication-bound），应优先优化 NCCL 拓扑}{Launch-bound（启动开销受限），应优先减少 Kernel 数量}

\qitem{4}{矩阵乘法 $C_{M\times N}=A_{M\times K}B_{K\times N}$ 是大模型中最核心的算子（Attention 投影、FFN 线性层等）。每个输出元素 $C_{ij}=\sum_{k=1}^{K}A_{ik}B_{kj}$ 需要 $K$ 次乘加（MAC，Multiply-Accumulate）。在 FLOPs 统计中，1 次 MAC 计为 2 次浮点运算（1 次乘法 + 1 次加法）。因此，完成整个 GEMM 所需的浮点运算量（FLOPs）为}{}
\fourchoicestwo{$MNK$（仅计乘法次数）}{$2MNK$}{$3MNK$}{$MN+NK+MK$（仅计矩阵元素总数）}

\qitem{5}{Google TPU 与 NVIDIA GPU 代表了两种不同的 AI 加速器设计哲学。GPU 采用 SIMT（单指令多线程）模型，擅长通用并行计算和复杂控制流；TPU 采用专用脉动阵列（Systolic Array），权重矩阵预加载到阵列中，激活值按固定节奏"流动"穿过阵列完成 MAC，数据流高度规整。TPU 在规整的大矩阵乘法上效率极高，但对不规则访存和控制流支持较弱。关于二者架构差异，下列说法\textbf{正确}的是}{}
\fourchoices{TPU 采用与 GPU 完全相同的 SIMT 模型，每个 TPU Core 可独立执行不同分支指令}{TPU 核心为二维脉动阵列（Systolic Array），数据在阵列中按固定节奏流动完成矩阵乘累加，无需像 GPU 那样为每个线程维护独立 PC}{GPU 的 Tensor Core 与 TPU 脉动阵列在硬件数据流上完全相同，只是 NVIDIA 的 marketing 命名不同}{NPU 是专有名词，仅指华为昇腾芯片，与 TPU、GPU 在架构上无任何可比性}

\qitem{6}{算子融合（Kernel Fusion）是编译器与手写 Kernel 中常用的优化手段：将多个连续算子（如 RMSNorm $\to$ Linear $\to$ SiLU）合并为单个 CUDA/Triton Kernel，使中间结果保留在寄存器或 Shared Memory 中，避免写回 HBM 再读取。融合还能减少 Kernel Launch 开销（每次 Launch 约 5--20$\mu$s 的 CPU 调度延迟）。工业界 vLLM、TensorRT-LLM 等均大量采用融合。关于算子融合的收益，下列说法\textbf{错误}的是}{}
\fourchoices{融合多个算子可减少 Kernel Launch 次数，降低 CPU 侧调度开销，对短算子链尤其明显}{融合后中间激活可保留在片上存储（寄存器/Shared Memory），减少 HBM 读写往返}{算子融合一定能将任意两个算子的总执行时间精确减半，且不受寄存器数量和算子形状限制}{RMSNorm + Linear、SiLU + Mul（SwiGLU 门控）是工业界最常见、收益最明显的融合模式之一}

\qitem{7}{W8A8 量化推理将权重和激活均量化为 INT8，在 INT32 累加器上完成 GEMM 后再反量化到 fp16/fp32。设激活 $\mathbf{X}$ 使用 per-tensor 缩放因子 $s_x$（整个张量共用一个 scale），权重 $\mathbf{W}$ 使用 per-channel 缩放因子 $s_w^{(j)}$（每个输出通道 $j$ 独立 scale，精度更高）。量化值为 $\mathbf{X}_{\mathrm{int8}}=\mathrm{round}(\mathbf{X}/s_x)$，$\mathbf{W}_{\mathrm{int8}}=\mathrm{round}(\mathbf{W}_{:,j}/s_w^{(j)})$。则等价的 fp32 线性层输出在工程实现中通常写为}{}
\fourchoices{$\mathbf{Y} = (\mathbf{X}_{\mathrm{int8}} \mathbf{W}_{\mathrm{int8}})$，直接输出 INT8 结果，无需反量化}{先执行 INT32 累加的 $\mathbf{Y}_{\mathrm{int32}}=\mathbf{X}_{\mathrm{int8}}\mathbf{W}_{\mathrm{int8}}$，再对每个输出通道 $j$ 乘 $s_x\cdot s_w^{(j)}$ 反量化到 fp16/fp32}{W8A8 严格要求权重和激活均使用 per-tensor 量化，不允许 per-channel}{INT8 GEMM 的累加必须在 INT8 域完成并直接输出，使用 INT32 累加器属于数值错误}

\qitem{8}{NVIDIA Tensor Core 自 Volta 架构引入，专门加速矩阵乘累加（MMA）。以 FP16 输入为例，标准 MMA 指令形状为 $m16n8k16$：每次处理 $16\times8$ 的输出 tile，沿 $K$ 维每次累加 16 个元素；输入为 FP16/BF16，累加器为 FP32 以保证精度。CUTLASS、cuBLASLt 等库通过多级 Tiling 将大 GEMM 分解为适配 MMA 形状的子块。关于 Tensor Core，下列说法\textbf{正确}的是}{}
\fourchoices{Tensor Core 仅支持 FP32 输入与 FP32 累加，与 CUDA Core 无区别}{标准 FP16 MMA 指令形状为 $m16n8k16$，输入 FP16/BF16、累加器 FP32；高效 GEMM 仍需配合多级 Tiling}{Tensor Core 与 CUDA Core 互斥，同一次 Kernel Launch 中不能混用二者}{调用 cuBLAS 的 GEMM 会自动使用 Tensor Core，开发者无需关心 Tiling 与数据布局}

\qitem{9}{GPTQ、AWQ、SmoothQuant 是大模型 PTQ（训练后量化）的三类代表性方法，其实现逻辑各不相同。GPTQ（Frantar et al., 2023）利用 Hessian 二阶信息逐列量化权重并传播误差；AWQ（Lin et al., 2024）识别对输出影响大的"显著权重通道"并按比例缩放；SmoothQuant（Xiao et al., 2023）通过可学习的对角矩阵 $S$ 将激活值中的离群点（Outlier）平滑迁移到权重侧，使激活更易量化。下列关于三者\textbf{实现逻辑}的匹配，\textbf{正确}的是}{}
\fourchoices{GPTQ 基于 Hessian 二阶信息逐列量化权重；AWQ 基于显著权重通道缩放；SmoothQuant 通过 $S$ 矩阵将激活离群点迁移到权重侧}{GPTQ 基于激活值平滑迁移；AWQ 基于 Hessian 逐列量化；SmoothQuant 基于显著通道缩放}{三者均属于 QAT（量化感知训练），训练时需插入 FakeQuant 算子并完整反向传播}{三者均只量化权重（Weight-Only），推理时激活值始终保持 fp32 不做任何压缩}

\qitem{10}{Triton（OpenAI）是面向 GPU 的高级 DSL，以 Block 为基本编程抽象：开发者描述"每个 Block 做什么"，编译器自动生成 Thread/Warp 索引、内存合并访问（Coalesced Access）和 Shared Memory 布局。相比之下，手写 CUDA 需显式管理 \texttt{threadIdx}、\texttt{blockIdx}、Warp 级原语（如 \texttt{\_\_shfl\_sync}）等底层细节。Triton 已广泛用于 FlashAttention、量化 GEMM 等自定义 Kernel。关于二者差异，下列说法\textbf{正确}的是}{}
\fourchoices{Triton 以 Thread 为基本编程单元，开发者需像 CUDA 一样手动计算每个线程的全局索引}{Triton 以 Block 为基本抽象，编译器自动处理 Thread/Warp 级并行与 Global Memory 合并访问}{Triton 无法编写自定义 GEMM Kernel，所有矩阵乘法必须调用 cuBLAS}{手写 CUDA 完全无法做算子融合，融合优化只能由 Triton 实现}

\qitem{11}{FlashAttention（Dao et al., NeurIPS 2022）通过 Tiling 将 $Q,K,V$ 分块加载到 SRAM，在片上完成 Online Softmax 并增量累加输出，避免将完整的 $N\times N$ 注意力矩阵 $S,P$ 写回 HBM。FlashAttention-2（Dao, 2023）进一步在 Warp 级别并行化、减少非矩阵乘 FLOPs，并优化了 SRAM 中的工作划分。关于 FlashAttention-2 的改进，下列\textbf{不包括}在内的是}{}
\fourchoices{减少非矩阵乘法的 FLOPs（如对 running statistic 的 rescale 操作进行合并与简化）}{在 Warp 级别更好地并行化 attention 计算，提升 SM 利用率}{将 $Q$、$K$、$V$ 的 SRAM 分块策略优化为跨 Warp 分工协作}{将注意力计算的 HBM IO 复杂度从 $O(N^2d^2/M)$ 提升到 $O(N^3)$，以换取更低的 SRAM 占用}

\qitem{12}{XLA（Google）和 TVM（Apache）是两类主流的深度学习编译器。XLA 将 TensorFlow/JAX 计算图降低为 HLO（High Level Optimizer）中间表示，执行算子融合（Fusion）、公共子表达式消除（CSE）、内存布局优化（Layout）等 Pass 后生成 LLVM/CUDA 代码。TVM 通过 AutoTVM/Ansor 对 Kernel 的 tile size、unroll factor、并行策略等超参进行自动搜索（Auto-tuning），在多种硬件后端上寻找最优实现。关于编译器优化，下列说法\textbf{正确}的是}{}
\fourchoices{XLA 的 HLO 图优化不支持算子融合，PyTorch Eager 模式下执行的算子无法被 XLA 优化}{TVM 通过 AutoTVM / Ansor 对 Kernel 的 tile size、unroll factor 等超参进行自动搜索，可面向 GPU/CPU 等多种后端}{TVM 是 Google 内部专用工具，只能为 TPU 生成代码，不支持 NVIDIA GPU}{XLA 的 Layout Optimization 与 Kernel Fusion 完全独立、互不影响，融合不会改变张量内存布局}

\qitem{19}{（本小题满分 10 分）}{}

\subsection*{\heiti 三、解答题答案}

\subsubsection*{19. 基础概念与 Roofline 计算（10 分）}

\noindent\textbf{(1) 存储层级（4 分）}

\begin{itemize}[leftmargin=2em,itemsep=0pt,parsep=0pt,topsep=0pt]
  \item \textbf{Register}：线程私有，速度最快、容量最小（每线程 KB 级）。
  \item \textbf{Shared Memory}：Block 内线程共享，SM 片上，需显式管理，速度接近 Register。
  \item \textbf{L2 Cache}：芯片级统一缓存，容量 MB 级，对程序员透明。
  \item \textbf{HBM}：容量最大（80GB），带宽高但延迟远高于片上存储。
\end{itemize}

\noindent\textbf{(2) Ridge Point（3 分）}

$I^*=\dfrac{989\times10^{12}}{3350\times10^9}\approx295$ FLOP/Byte。物理含义：$I<295$ 时吞吐 $=I\cdot B_w$，受带宽限制；$I>295$ 时吞吐 $\approx 989$ TFLOPS，受算力限制。

\noindent\textbf{(3) 吞吐估算（3 分）}

$I=10$ FLOP/Byte $\ll 295$，Memory-bound。可达吞吐 $\approx I\cdot B_w=10\times3350=33500$ GB/s$\times$... 注意单位：$I\cdot B_w=10\times3350$ GFLOPS $=33.5$ TFLOPS（保留整数约 \textbf{34 TFLOPS}）。

\subsubsection*{20. KV Cache 与 Decode 带宽（10 分）}

\noindent\textbf{(1) $M_{\mathrm{KV}}$（3 分）}

$M_{\mathrm{KV}}=2\times80\times4096\times8\times128\times2=2^{31}$ 字节 $=2$ GB。

\noindent\textbf{(2) 每步读取与 Memory-bound（3 分）}

每步读取全部 KV：约 2 GB。每步 FLOPs 仅 $O(d)\sim$ 数万，算术强度 $\sim 0.01$--$1$ FLOP/Byte $\ll 295$，故 Decode 为 Memory-bound。

\noindent\textbf{(3) fp8 KV（4 分）}

fp8 每标量 1 字节，$M_{\mathrm{KV}}$ 减半为 1 GB。HBM 读取量减半，在同样算力下有效带宽压力降低，工作点沿 Roofline 向左上方移动（等效提升吞吐）。

\subsubsection*{21. FlashAttention 公式与图示（10 分）}

\noindent\textbf{(1) 图示分析（3 分）}

标准 Attention 将 $N\times N$ 的 $S,P$ 写回 HBM 再读取，IO 量 $O(N^2)$，长序列时带宽成为瓶颈。FlashAttention 在 SRAM 分块计算 Online Softmax，$S,P$ 不写出，仅 $O$ 最终结果写回。

\noindent\textbf{(2) rescale 项（4 分）}

$m'=\max(m,m^{(j)})$ 时若最大值更新，旧块的 softmax 归一化分母 $\ell$ 和分子贡献 $O$ 需按 $e^{m-m'}$ 重新缩放，以保证合并新块后全局 softmax 正确。

\noindent\textbf{(3) 复杂度优势（3 分）}

标准 Attention HBM IO 至少 $O(N^2)$（写出 $S,P$）。FlashAttention 为 $O(N^2d^2/M_{\mathrm{sram}})$，$M_{\mathrm{sram}}$ 为常数时随 $N$ 增长更慢，长序列优势显著。

\subsubsection*{22. 量化公式应用（10 分）}

\noindent\textbf{(1) $s_3$（3 分）}

$s_3=\dfrac{32^{0.5}}{0.5^{0.5}}=\dfrac{\sqrt{32}}{\sqrt{0.5}}=\sqrt{64}=8.00$。$\alpha=0.5$ 时激活与权重各承担一半量化难度。

\noindent\textbf{(2) 等价性与易量化（3 分）}

$(\mathbf{X}S)(S^{-1}\mathbf{W})=\mathbf{X}(SS^{-1})\mathbf{W}=\mathbf{XW}$。$\mathbf{X}S$ 将大绝对值离群点除以 $s_j$ 压缩，动态范围变小，INT8 量化误差降低。

\noindent\textbf{(3) GPTQ 更新（4 分）}

$w_4\leftarrow 1.0-\dfrac{0.1}{1.0}\times(-0.02)=1.0+0.002=1.002$。

\subsubsection*{23. PagedAttention 架构设计（10 分）}

\noindent\textbf{(1) 碎片问题（3 分）}

固定 Block（16 tokens）分配消除内部碎片；Block Table 将逻辑连续 KV 映射到非连续物理 Block，释放的 Block 可立即复用，消除外部碎片。

\noindent\textbf{(2) Block 数量与总显存（4 分）}

单请求 Block 数 $=\lceil 4096/16\rceil=256$。单请求 KV $\approx 2$ GB，100 并发 $\approx 200$ GB（需多卡或量化/卸载）。

\noindent\textbf{(3) 调度策略（3 分）}

预分配 Block 池；长请求（32k）限制并发数或 KV Offload；短请求优先；OOM 时按 LRU 抢占/换出低优先级请求 KV。

\subsubsection*{24. H100 综合架构设计（10 分）}

\noindent\textbf{(1) Roofline 分析（3 分）}

\texttt{dram\_throughput} 高、\texttt{sm\_\_throughput} 低 $\Rightarrow$ Memory-bound。Decode 算术强度约 0.1--2 FLOP/Byte，远低于 $I^*=295$。

\noindent\textbf{(2) 优化栈（4 分）}

\begin{itemize}[leftmargin=2em,itemsep=0pt,parsep=0pt,topsep=0pt]
  \item \textbf{W8A8/FP8 KV}：减半 HBM 读取 $\to$ Roofline 上垂直上移。
  \item \textbf{Kernel Fusion}：减少中间写回 $\to$ 提高有效 $I$ $\to$ 工作点右移。
  \item \textbf{Continuous Batching}：增大有效 Batch $\to$ GEMM $M$ 维增大 $\to$ $I$ 提升。
  \item \textbf{FlashAttention-2}：优化 attention 带宽利用。
\end{itemize}

\noindent\textbf{(3) 验收指标（3 分）}

\texttt{sm\_\_throughput} 应上升；\texttt{dram\_bytes\_read}/token 应下降；\texttt{achieved\_occupancy} 应提升。
